# Лабораторная работа №6  
## Обработка и валидация форм на PHP

## Описание проекта

Данный проект выполнен в рамках лабораторной работы №6 по теме **«Обработка и валидация форм»**.

Проект представляет собой простое веб-приложение на PHP для ведения **трекера привычек**. Пользователь может добавить новую привычку через HTML-форму, после чего данные проходят серверную проверку и сохраняются в файл `data.txt` в формате JSON. Также реализована отдельная страница для просмотра всех сохранённых записей в виде таблицы.

Главная цель проекта — показать работу с HTML-формами, методом POST, обработкой данных на сервере, валидацией пользовательского ввода и сохранением информации в файл.

---

## Цель работы

Освоить основные принципы работы с HTML-формами в PHP, включая:

- отправку данных на сервер;
- получение данных через `$_POST`;
- валидацию пользовательского ввода;
- сохранение данных в файл;
- вывод сохранённых записей;
- защиту данных при отображении на странице.

## Назначение файлов

### `index.php`

`index.php` — это главная страница проекта.

В этом файле находится HTML-форма для добавления новой привычки. Пользователь вводит название привычки, описание, категорию, приоритет, дату начала, время напоминания и статус.

Форма отправляет данные в файл `save.php` методом `POST`.

```html
<form action="save.php" method="POST">
```

В форме используются элементы:

- текстовое поле для названия привычки;
- поле `textarea` для описания;
- выпадающий список для категории;
- переключатели `radio` для приоритета;
- поле выбора даты;
- поле выбора времени;
- выпадающий список для статуса.

Также в форме добавлена клиентская валидация с помощью HTML-атрибутов:

- `required`;
- `minlength`;
- `maxlength`;
- `type="date"`;
- `type="time"`.

Это помогает пользователю сразу увидеть ошибки ещё до отправки формы на сервер.

---

### `save.php`

`save.php` — это серверный обработчик формы.

Этот файл получает данные, которые пользователь отправил из `index.php`, проверяет их и сохраняет в файл `data.txt`.

Основные действия файла:

1. Подключает классы валидаторов.
2. Проверяет, что форма была отправлена методом `POST`.
3. Получает данные из массива `$_POST`.
4. Очищает введённые значения.
5. Проверяет каждое поле через валидаторы.
6. Если есть ошибки, показывает их пользователю.
7. Если ошибок нет, создаёт новую запись.
8. Сохраняет запись в `data.txt` в формате JSON.
9. Показывает сообщение об успешном сохранении.

Для получения данных используется массив:

```php
$_POST
```

Например:

```php
$_POST['title'] ?? ''
```

Это значит: взять значение поля `title`, а если его нет — использовать пустую строку.

Данные перед сохранением очищаются, чтобы убрать лишние пробелы и HTML-теги.

---

### `list.php`

`list.php` — это страница для просмотра всех сохранённых привычек.

Этот файл читает данные из `data.txt`, преобразует JSON в массив PHP и выводит записи в виде HTML-таблицы.

Файл выполняет следующие задачи:

- читает файл `data.txt`;
- преобразует JSON-данные в массив;
- проверяет наличие записей;
- выводит данные в таблице;
- позволяет сортировать записи;
- безопасно отображает пользовательский текст.

Для защиты от XSS при выводе используется функция:

```php
htmlspecialchars()
```

Она нужна для того, чтобы введённый пользователем текст отображался как обычный текст, а не выполнялся как HTML или JavaScript-код.

---

### `ValidatorInterface.php`

`ValidatorInterface.php` — это интерфейс для всех валидаторов.

Он задаёт единый стандарт для классов проверки данных. Благодаря этому все валидаторы имеют одинаковую структуру и могут использоваться в одном общем механизме проверки.

Интерфейс содержит два метода:

```php
validate($value): bool
```

Метод `validate()` проверяет значение и возвращает:

- `true`, если значение корректное;
- `false`, если значение некорректное.

Также есть метод:

```php
getErrorMessage(): string
```

Он возвращает текст ошибки, если проверка не прошла.

Этот файл важен для архитектуры проекта, потому что делает систему валидации более удобной, понятной и расширяемой.

---

### `RequiredValidator.php`

`RequiredValidator.php` — это валидатор обязательного поля.

Он проверяет, что пользователь не оставил поле пустым.

Например, если поле названия привычки не заполнено, валидатор вернёт ошибку.

Примеры значений:

```text
""        — ошибка
"   "     — ошибка
"Спорт"   — корректно
```

Этот валидатор используется почти для всех полей формы, потому что большинство данных являются обязательными для заполнения.

---

### `LengthValidator.php`

`LengthValidator.php` — это валидатор длины строки.

Он проверяет, что текст не меньше минимального значения и не больше максимального значения.

Например:

- название привычки должно быть от 3 до 100 символов;
- описание должно быть от 10 до 500 символов.

Для подсчёта длины используется функция:

```php
mb_strlen()
```

Она подходит для русского текста, потому что корректно считает символы кириллицы.

---

### `DateValidator.php`

`DateValidator.php` — это валидатор даты.

Он проверяет, что дата введена в правильном формате:

```text
YYYY-MM-DD
```

Пример правильной даты:

```text
2026-04-09
```

Примеры неправильных дат:

```text
09.04.2026
2026/04/09
2026-02-31
```

Для проверки используется механизм PHP `DateTime::createFromFormat()`.

---

### `TimeValidator.php`

`TimeValidator.php` — это валидатор времени.

Он проверяет, что время указано в формате:

```text
HH:MM
```

Примеры правильного времени:

```text
09:30
14:45
23:59
```

Примеры неправильного времени:

```text
25:90
9:5
99:99
```

Проверка выполняется с помощью регулярного выражения, которое контролирует часы и минуты.

---

### `EnumValidator.php`

`EnumValidator.php` — это валидатор для полей с ограниченным набором значений.

Он проверяет, что пользователь отправил только одно из разрешённых значений.

Например, для поля `priority` разрешены только:

```text
low
medium
high
```

Если пользователь попытается отправить другое значение, валидатор вернёт ошибку.

Это важно, потому что данные нельзя проверять только на стороне HTML-формы. Пользователь может изменить код страницы или отправить запрос вручную, поэтому серверная проверка обязательна.

---

### `data.txt`

`data.txt` — это файл для хранения данных.

В этом проекте вместо базы данных используется обычный текстовый файл в формате JSON.

Изначально файл содержит пустой массив:

```json
[]
```

После добавления привычек файл может выглядеть так:

```json
[
    {
        "title": "Пить воду",
        "description": "Выпивать 2 литра воды каждый день",
        "category": "health",
        "priority": "high",
        "start_date": "2026-04-09",
        "reminder_time": "09:00",
        "status": "active",
        "created_at": "2026-04-09"
    }
]
```

Такой формат удобен для лабораторной работы, потому что:

- не нужно подключать базу данных;
- данные легко читать;
- JSON удобно преобразовывать в массив PHP;
- файл можно открыть и проверить вручную.

Общая схема работы проекта:

```text
index.php
   ↓
save.php
   ↓
ValidatorInterface.php
RequiredValidator.php
LengthValidator.php
DateValidator.php
TimeValidator.php
EnumValidator.php
   ↓
data.txt
   ↓

```

Пользователь сначала открывает `index.php` и заполняет форму. После отправки данные переходят в `save.php`, где проходят очистку и проверку. Для проверки используются отдельные валидаторы. Если данные корректные, они сохраняются в `data.txt`. Затем `list.php` читает этот файл и показывает все записи пользователю в таблице.
## Контрольные вопросы

### Какие существуют методы отправки данных из формы на сервер? Какие методы поддерживает HTML-форма?

HTML-форма может отправлять данные на сервер разными способами, но чаще всего используются методы `GET` и `POST`.

Метод `GET` передаёт данные через адресную строку браузера. Такой способ удобно использовать для поиска, фильтрации, сортировки и других действий, где данные не являются секретными.

Пример формы с методом `GET`:

```html
<form action="list.php" method="GET">
```

Метод `POST` передаёт данные в теле HTTP-запроса. Он чаще используется для отправки форм регистрации, авторизации, добавления записей и других действий, при которых данные не должны отображаться в ссылке.

Пример формы с методом `POST`:

```html
<form action="save.php" method="POST">
```

В данной лабораторной работе используется метод `POST`, потому что пользователь добавляет новую привычку, а данные должны отправляться на сервер для обработки и сохранения.

---

### Какие глобальные переменные используются для доступа к данным формы в PHP?

В PHP для доступа к данным, отправленным из формы, используются суперглобальные массивы.

Основные массивы:

```php
$_GET
$_POST
$_REQUEST
```

Массив `$_GET` используется тогда, когда форма отправляет данные методом `GET`.

Например:

```php
$title = $_GET['title'] ?? '';
```

Массив `$_POST` используется тогда, когда форма отправляет данные методом `POST`.

Например:

```php
$title = $_POST['title'] ?? '';
```

Массив `$_REQUEST` может содержать данные из `$_GET`, `$_POST` и `$_COOKIE`, но на практике лучше использовать именно `$_GET` или `$_POST`. Так код становится понятнее, безопаснее и легче проверяется.

В данной лабораторной работе используется массив `$_POST`, потому что форма отправляет данные методом `POST` в файл `save.php`.

---

### Как обеспечить безопасность при обработке данных из формы?

При обработке данных из формы нельзя полностью доверять информации, которую вводит пользователь. Пользователь может случайно ввести неправильные данные или специально отправить вредный код.

Поэтому данные нужно:

- проверять;
- очищать;
- безопасно выводить на страницу.

Для очистки данных можно использовать функции:

```php
trim()
strip_tags()
```

Функция `trim()` убирает лишние пробелы в начале и конце строки.

Функция `strip_tags()` удаляет HTML-теги из введённого текста.

Пример очистки данных:

```php
$title = trim(strip_tags($_POST['title'] ?? ''));
```

Для защиты от XSS при выводе данных на страницу используется функция:

```php
htmlspecialchars()
```

Она превращает HTML-теги в обычный текст, поэтому вредный JavaScript-код не выполняется в браузере.

Пример безопасного вывода:

```php
echo htmlspecialchars($title, ENT_QUOTES, 'UTF-8');
```

Также важно выполнять серверную валидацию: проверять обязательные поля, длину текста, формат даты, формат времени и допустимые значения для полей с ограниченным выбором.

В данной лабораторной работе безопасность обеспечивается за счёт очистки данных, проверки через валидаторы и безопасного вывода через `htmlspecialchars()`.
