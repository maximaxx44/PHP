Задание 1. Анализ HTTP-запросов. Часть 1

Откройте сайт Wikipedia: https://en.wikipedia.org/wiki/HTTP

Прочитайте статью и обратите внимание на разделы, посвященные HTTP-запросам.

Откройте инструменты разработчика в вашем браузере (обычно это можно сделать, нажав F12 или правой кнопкой мыши выбрав "Inspect").

Перейдите на вкладку "Network" (Сеть).

Обновите страницу Wikipedia.

Найдите первый запрос к странице, он будет назван как HTTP и изучите следующие аспекты:

URL запроса

URL запроса: https://en.wikipedia.org/wiki/HTTP

Метод запроса. Почему именно данный метод используется для загрузки страницы?

Метод запроса: GET. Он используется потому что браузер запрашивает страницу для чтения. GET это стандартный метод HTTP для запроса ресурса без изменения его на сервере.

Статус ответа. Что означает статус ответа?

Статус ответа: 200 OK. Это значит, что все прошло успешно. 200 OK - стандартный код успеха, сервер вернул страницу без ошибок.

Заголовки запроса и ответа. Какие заголовки присутствуют и какую информацию они содержат?

Заголовки запроса включают параметры, которые браузер отправляют серверу. В этом случае присутствуют заголовки: :authority (означает домен и содержит en.wikipedia.org), user-agent (браузер и устройство, содержит Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/144.0.0.0 Safari/537.36), accept (показывает какие типы данных браузер готов принять, содержит text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,image/apng,*/*;q=0.8,application/signed-exchange;v=b3;q=0.7), accept-language (предпочтительный язык, содержит ru-RU,ru;q=0.9,en-US;q=0.8,en;q=0.7)

Заголовки ответа: content-type (означает, что это HTML-страница, содержит text/html; charset=UTF-8), date (содержит дату и время сервера), server (содержит информацию о сервере), content-length (размер тела ответа), cache-control (содержит информацию о том, как кешировать страницу), last-modified (данные для кеширования), strict-transport-security (принудительное HTTPS)

Есть ли тело запроса или ответа? Если да, то что в нем содержится?

Для URL https://en.wikipedia.org/wiki/HTTP при загрузке страницы используется метод GET, поэтому тело запроса отсутствует - браузер просто запрашивает страницу у сервера. В теле ответа сервер возвращает HTML-код статьи Wikipedia “HTTP”, который содержит структуру страницы, текст статьи, ссылки, таблицы и другие элементы. Именно это содержимое браузер затем отображает пользователю как готовую веб-страницу.

Какие еще запросы были отправлены при загрузке страницы и почему?

При загрузке страницы браузер также загружает CSS файлы для оформления статьи JavaScript для работы интерфейса Wikipedia, а также изображения и иконки, которые используются на странице. Эти запросы нужны потому, что сама HTML-страница содержит только структуру и текст, а внешний вид, стили и дополнительные элементы загружаются отдельно. Поэтому в Network видно не один запрос, а много - каждый отвечает за свою часть отображения страницы.

Задание 2. Анализ HTTP-запросов. Часть 2

Перейдите на: https://en.wikipedia.org/wiki/Special:Search

Выполните поиск по слову browser (Пример).

Найдите запрос поиска в Network.

Изучите следующие аспекты:

URL запроса. Какой URL используется для выполнения поиска?

Для выполнения поиска используется URL https://en.wikipedia.org/w/index.php?search=browser&title=Special%3ASearch&profile=advanced&fulltext=1&ns0=1

Метод запроса. Почему используется именно этот метод?

Метод - GET. Он используется потому что происходит получение информации с сервера, а не изменение данных, параметры поиска удобно передавать в URL, такую ссылку можно сохранить или отправить другому человеку.

Query Parameters. Какие параметры передаются в запросе, что они означают и для чего используются?

После символа ? находятся параметры поиска: search (слово, которое вводит пользователь), title (указывает, что используется страница поиска), ns0 (поиск только по основным статьям Wikipedia), profile=advanced (означает, что используется расширенный поиск), fulltext=1 (поиск не только по заголовкам, но и по содержимому статей).

Задание 4. Составление HTTP-запросов

Составьте GET-запрос к серверу по адресу http://sandbox.usm.com, указав в заголовке User-Agent ваше имя и фамилию.

GET / HTTP/1.1

Host: sandbox.usm.com

User-Agent: Maxim Orlov

Accept: */*

Connection: close

Что такое User-Agent и для чего он используется?

User-Agent - это HTTP-заголовок, который сообщает серверу, какой клиент отправляет запрос (браузер, приложение, устройство и т.д.). Он используется сервером для анализа трафика, адаптации контента под устройство пользователя и диагностики работы сайта.

Составьте POST-запрос к серверу по адресу http://sandbox.usm.com/cars, указав в теле запроса следующие параметры: make: марка автомобиля (например, "Toyota") model: модель автомобиля (например, "Corolla") year: год выпуска автомобиля (например, 2020)

POST /cars HTTP/1.1

Host: sandbox.usm.com

User-Agent: Maxim Orlov

Content-Type: application/x-www-form-urlencoded

Connection: close

make=Toyota&model=Corolla&year=2020

Какие еще методы HTTP-запросов существуют и для чего они используются?

GET - получить ресурс (страницу/данные).

POST - отправить данные на сервер, обычно создать ресурс/запустить обработку.

PUT - заменить ресурс целиком (полное обновление).

PATCH - частично обновить ресурс (изменить только поля).

DELETE - удалить ресурс.

HEAD - как GET, но без тела ответа (только заголовки).

OPTIONS - узнать, какие методы/настройки доступны (часто для CORS).

CONNECT - установить туннель (часто для HTTPS через прокси).

TRACE - диагностика (редко используется, часто отключен).

Составьте PUT-запрос к серверу по адресу http://sandbox.usm.com/cars/1, указав в заголовке User-Agent ваше имя и фамилию, в заголовке Content-Type значение application/json и в теле запроса следующие параметры:

{
 "make": "Toyota",
 
 "model": "Corolla",
 
 "year": 2021
}

PUT /cars/1 HTTP/1.1

Host: sandbox.usm.com

User-Agent: Maxim Orlov

Content-Type: application/json

Connection: close

{

  "make": "Toyota",
  
  "model": "Corolla",
  
  "year": 2021
  
}

В чем разница между PATCH и PUT запросами?

PUT обычно означает полную замену ресурса: пользователь отправляет “полную новую версию” объекта. Если не отправил какое-то поле, сервер может считать его отсутствующим/сбросить (зависит от API). PATCH означает частичное изменение: отправляешь только то, что меняешь. Например, поменять только year, не трогая make и model.

Пример PATCH:

PATCH /cars/1 HTTP/1.1

Host: sandbox.usm.com

Content-Type: application/json

{"year":2022}

Предположите ситуации, когда сервер на запрос выше может вернуть HTTP-коды состояния 200, 201, 400, 401, 403, 404, 500.

200 OK:

Может быть возвращен, если сервер успешно обработал запрос и возвращает результат операции, но не создает новый ресурс (например, обновляет существующий).

 201 Created:

Возвращается, если автомобиль успешно создан и добавлен в систему. Обычно сопровождается заголовком Location с адресом созданного ресурса.

 400 Bad Request:

Возвращается, если запрос содержит ошибки:

неверный формат данных,

отсутствуют обязательные поля,

неправильный Content-Type.

 401 Unauthorized:

Возвращается, если требуется аутентификация, но пользователь не прошел ее (например, отсутствует токен или сессия).

 403 Forbidden:

Возвращается, если пользователь аутентифицирован, но не имеет прав на создание ресурса.

 404 Not Found:

Может быть возвращён, если указанный путь /cars не существует на сервере.

 500 Internal Server Error:

Возвращается при внутренней ошибке сервера, например:

сбой базы данных,

ошибка обработки данных,

исключение в серверном коде.
