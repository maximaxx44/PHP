# Лабораторная работа №7  
## Шаблонизация в PHP

## Описание проекта

Данный проект выполнен в рамках лабораторной работы №7 по теме **«Шаблонизация»**.

Проект является продолжением лабораторной работы №6, где была реализована форма для добавления привычек, их обработка, валидация и сохранение данных в файл.

В этой работе проект был переработан так, чтобы отделить **логику приложения** от **представления**. Также были реализованы два варианта шаблонизации:

1. нативные PHP-шаблоны;
2. шаблоны с использованием Twig.

Главная цель работы — улучшить структуру проекта, сделать код более понятным, удобным для поддержки и расширения.

---

## Цель работы

Освоить принципы шаблонизации в PHP:

- с использованием нативных PHP-шаблонов;
- с использованием готового шаблонизатора Twig;
- с разделением логики и представления;
- с применением наследования шаблонов;
- с использованием блоков, include и фильтров.

---

## Условие работы

Необходимо продолжить разработку проекта из лабораторной работы №6 и разделить его на два уровня:

- **логика** — PHP-файлы, которые обрабатывают запросы, читают и записывают данные;
- **представление** — шаблоны, которые только отображают подготовленные данные.

Работа выполняется в два этапа:

1. реализация собственного механизма шаблонизации через нативные PHP-шаблоны;
2. подключение Twig и реализация той же функциональности через готовый шаблонизатор.

## Разделение проекта на логику и представление

В проекте появились два основных уровня:

### Логика

К логике относятся файлы, которые отвечают за обработку данных:

```text
src/functions.php
src/handler.php
index.php
list.php
```

Эти файлы выполняют следующие задачи:

- принимают данные из формы;
- проверяют запросы;
- читают данные из файла;
- сохраняют новые записи;
- подготавливают переменные для шаблонов.

---

### Представление

К представлению относятся файлы из папки `templates/`:

```text
templates/layout.php
templates/form.php
templates/list.php
```

Эти файлы не должны заниматься обработкой данных.  
Их задача — только отображать уже подготовленные переменные.

Такой подход делает проект более аккуратным, потому что HTML-код не смешивается с основной логикой приложения.

---

## Назначение файлов

### `src/functions.php`

Файл `src/functions.php` содержит вспомогательные функции проекта.

В него были вынесены функции для работы с данными:

- чтение записей из файла `data.txt`;
- сохранение новых записей;
- подготовка данных для вывода;
- дополнительные функции, которые используются в разных частях проекта.

Этот файл относится к логике проекта, потому что он не выводит HTML-страницу, а только помогает работать с данными.

---

### `src/handler.php`

Файл `src/handler.php` отвечает за обработку формы.

Он проверяет, был ли отправлен `POST`-запрос. Если форма была отправлена, файл считывает данные, проверяет введённые значения и сохраняет новую привычку в `data.txt`.

Основные задачи файла:

1. проверить метод запроса;
2. получить данные из формы;
3. выполнить базовую проверку;
4. подготовить новую запись;
5. сохранить запись в файл;
6. передать результат дальше.

---

### `templates/layout.php`

Файл `templates/layout.php` содержит общий макет страницы.

В него вынесены повторяющиеся части HTML-документа:

- `DOCTYPE`;
- тег `html`;
- блок `head`;
- подключение стилей;
- шапка сайта;
- основная область страницы;
- общая структура вывода.

Этот файл нужен для того, чтобы не писать один и тот же HTML-код на каждой странице.

---

### `templates/form.php`

Файл `templates/form.php` содержит форму добавления привычки.

Именно в этом шаблоне находится HTML-разметка формы:

```html
<form method="POST">
```

Форма позволяет пользователю ввести данные новой привычки и отправить их на сервер.

В шаблон вынесена только разметка формы.  
Обработка отправленных данных выполняется не здесь, а в файлах логики.

---

### `templates/list.php`

Файл `templates/list.php` отвечает за вывод списка привычек.

Он получает уже подготовленный массив данных и отображает его в виде таблицы или списка.

В этом шаблоне могут использоваться PHP-вставки для вывода переменных, например:

```php
<?= htmlspecialchars($habit['title']) ?>
```

Такой вывод помогает безопасно показать пользовательские данные на странице.

---

### `index.php`

Файл `index.php` является точкой входа для страницы с формой.

Он подключает необходимые файлы, подготавливает переменные и передаёт их в шаблон.

Его задача — не хранить всю HTML-разметку внутри себя, а использовать отдельные шаблоны из папки `templates/`.

---

### `list.php`

Файл `list.php` отвечает за страницу со списком привычек.

Он получает данные из `data.txt`, подготавливает их и подключает шаблон списка.

Таким образом, `list.php` относится к логике страницы, а отображение данных выполняется в `templates/list.php`.

## Этап 2. Использование шаблонизатора Twig

На втором этапе в проект был добавлен готовый шаблонизатор **Twig**.

Twig позволяет писать шаблоны с более чистым и удобным синтаксисом.  
Он помогает отделить HTML-разметку от PHP-кода и предоставляет дополнительные возможности:

- наследование шаблонов;
- блоки;
- подключение отдельных шаблонов;
- фильтры;
- безопасный вывод переменных.

---

## Подключение Twig

Для подключения Twig был создан файл `composer.json`.

В нём была указана зависимость:

```json
{
    "require": {
        "twig/twig": "^3.0"
    }
}
```

После этого была выполнена команда:

```bash
composer install
```

Composer установил Twig и создал папку `vendor/`, в которой находятся файлы библиотеки и автозагрузчик.

---

## Структура Twig-шаблонов

Для Twig-версии проекта была создана отдельная папка:

```text
twig_templates/
```

В неё были помещены следующие шаблоны:

```text
twig_templates/
│
├── layout.twig
├── index.twig
├── form.twig
└── list.twig
```

---

## Назначение Twig-файлов

### `twig_templates/layout.twig`

Файл `layout.twig` содержит общий макет страницы.

В нём находится основная HTML-структура сайта:

- начало HTML-документа;
- блок `head`;
- заголовок страницы;
- общая структура страницы;
- место для основного содержимого.

В этом файле используется механизм блоков Twig:

```twig
{% block content %}{% endblock %}
```

Блок `content` является местом, куда дочерние шаблоны вставляют своё содержимое.

---

### `twig_templates/index.twig`

Файл `index.twig` отвечает за страницу с формой.

Он наследует общий макет с помощью конструкции:

```twig
{% extends "layout.twig" %}
```

Затем он заполняет блок `content`:

```twig
{% block content %}
    {% include "form.twig" %}
{% endblock %}
```

То есть страница использует общий макет, а внутрь основной части подключает форму из файла `form.twig`.

---

### `twig_templates/form.twig`

Файл `form.twig` содержит форму добавления привычки.

Он отвечает только за отображение HTML-формы.  
Логика обработки данных в этом файле не выполняется.

Такой подход делает шаблон чистым и понятным.

---

### `twig_templates/list.twig`

Файл `list.twig` отвечает за вывод списка привычек.

Он получает данные из PHP-файла и отображает их пользователю.

В Twig переменные выводятся через двойные фигурные скобки:

```twig
{{ habit.title }}
```

Также можно использовать фильтры:

```twig
{{ habit.status|status_icon }}
```

---

## Файлы запуска Twig-версии

Для запуска Twig-версии были созданы отдельные файлы:

```text
index_twig.php
list_twig.php
```

---

### `index_twig.php`

Файл `index_twig.php` отвечает за страницу с формой в Twig-версии проекта.

В нём подключается автозагрузчик Composer:

```php
require_once __DIR__ . '/vendor/autoload.php';
```

Затем создаётся загрузчик шаблонов:

```php
$loader = new FilesystemLoader(__DIR__ . '/twig_templates');
```

После этого создаётся объект Twig:

```php
$twig = new Environment($loader);
```

Далее данные передаются в шаблон через метод `render()`:

```php
echo $twig->render('index.twig', [
    'pageTitle' => 'Трекер привычек',
    'habits' => $habits
]);
```

---

### `list_twig.php`

Файл `list_twig.php` отвечает за вывод списка привычек через Twig.

Он получает данные из файла `data.txt`, подготавливает массив привычек и передаёт его в Twig-шаблон `list.twig`.

Это позволяет отобразить список привычек без смешивания PHP-логики и HTML-разметки.

## Дополнительная функция

Для получения максимальной оценки был реализован собственный фильтр Twig.

Фильтр называется:

```text
status_icon
```

Он используется для отображения статуса привычки в более понятном виде.

Пример использования в Twig-шаблоне:

```twig
{{ habit.status|status_icon }}
```

Фильтр преобразует текстовый статус в значок.

Например:

```text
done     → ✅
not_done → ❌
default  → ➖
```

---

## Регистрация собственного фильтра Twig

Фильтр регистрируется в PHP-файле Twig-версии проекта.

Пример регистрации:

```php
$twig->addFilter(new TwigFilter('status_icon', function ($status) {
    return match ($status) {
        'done' => '✅',
        'not_done' => '❌',
        default => '➖'
    };
}));
```

Этот фильтр делает список привычек более наглядным для пользователя.

Вместо обычного текстового статуса пользователь видит понятный значок.

## Контрольные вопросы

### 1. В чём отличие нативных PHP-шаблонов от шаблонизатора Twig?

Нативные PHP-шаблоны — это обычные `.php`-файлы, в которых HTML-разметка может сочетаться с небольшими вставками PHP-кода.

Например:

```php
<?= htmlspecialchars($title) ?>
```

Twig — это отдельный шаблонизатор со своим синтаксисом.  
В нём переменные выводятся так:

```twig
{{ title }}
```

Главное отличие в том, что PHP-шаблоны не требуют установки дополнительных библиотек, но в них легче случайно смешать логику и разметку.

Twig делает шаблоны чище и удобнее. Он поддерживает наследование, блоки, фильтры и подключение отдельных шаблонов. Однако для его использования нужно установить библиотеку через Composer и изучить синтаксис Twig.

---

### 2. Зачем разделять логику и представление в проекте?

Разделение логики и представления нужно для того, чтобы код был более понятным, аккуратным и удобным для поддержки.

Логика отвечает за обработку данных:

- получение данных из формы;
- проверку значений;
- чтение файла;
- сохранение информации;
- подготовку переменных.

Представление отвечает только за внешний вид страницы:

- HTML-разметку;
- вывод переменных;
- таблицы;
- формы;
- кнопки;
- ссылки.

Если смешивать всё в одном файле, проект быстро становится запутанным. Сложнее искать ошибки, менять дизайн и добавлять новые функции.

Когда логика и представление разделены, можно изменить внешний вид страницы, почти не затрагивая обработку данных.

---

### 3. Что такое наследование шаблонов в Twig?

Наследование шаблонов в Twig — это механизм, который позволяет создать один общий шаблон и использовать его на разных страницах.

Например, можно создать файл:

```text
layout.twig
```

В нём находится общая структура сайта:

- HTML-документ;
- заголовок;
- подключение стилей;
- шапка;
- основная область страницы.

Внутри общего шаблона создаётся блок:

```twig
{% block content %}{% endblock %}
```

А дочерний шаблон подключает общий макет с помощью команды:

```twig
{% extends "layout.twig" %}
```

После этого дочерний шаблон может заполнить блок своим содержимым:

```twig
{% block content %}
    {% include "form.twig" %}
{% endblock %}
```

Это удобно, потому что не нужно повторять один и тот же HTML-код на каждой странице. Общая структура хранится в одном месте, а разные страницы меняют только своё основное содержимое.
