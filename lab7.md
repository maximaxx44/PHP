Лабораторная работа №7. Шаблонизация

Цель работы

Освоить принципы шаблонизации в PHP, как с использованием нативных PHP-шаблонов, так и с применением готового шаблонизатора. Улучшить структуру проекта, разделив логику обработки данных и представление.

Условие

Продолжите разработку проекта из лабораторной работы №6, разделив его на слои: логику и представление. Работа выполняется в два этапа - сначала вы реализуете собственный механизм шаблонизации, затем подключаете Twig и реализуете ту же функциональность через него.

Шаг 1. Рефакторинг. Нативные PHP-шаблоны

Перепишите существующий проект, разделив код на два уровня:

Логика - PHP-файлы, которые обрабатывают запросы, читают/записывают данные и подготавливают переменные.

Представление - отдельные .php-файлы-шаблоны, которые только отображают данные.

На первом этапе проект из лабораторной работы №6 был переработан.

Главная задача заключалась в том, чтобы отделить код, который отвечает за обработку данных, от кода, который отвечает за отображение страницы.

Была создана следующая структура проекта:

lab7_habit_tracker/

├── src/

│   ├── functions.php

│   └── handler.php

├── templates/

│   ├── layout.php

│   ├── form.php

│   └── list.php

├── index.php

├── list.php

└── data.txt

Папка src/ содержит логику проекта.

В файл src/functions.php были вынесены функции для работы с данными: чтение записей из файла data.txt, сохранение новых записей и подготовка данных для вывода.

Файл src/handler.php отвечает за обработку формы. В нём проверяется, был ли отправлен POST-запрос, считываются данные из формы, выполняется проверка введённых значений, после чего новая привычка сохраняется в файл data.txt.

Папка templates/ содержит нативные PHP-шаблоны. Эти файлы не выполняют обработку данных, а только отображают уже подготовленные переменные.

В шаблон templates/layout.php был вынесен общий макет страницы: структура HTML-документа, заголовок, подключение стилей, шапка и основная область страницы.

В шаблон templates/form.php была вынесена форма добавления привычки.

В шаблон templates/list.php был вынесен вывод списка привычек.

Таким образом, в проекте появились два уровня:

Логика:

src/functions.php

src/handler.php

index.php

list.php

Представление:

templates/layout.php

templates/form.php

templates/list.php

Файл index.php стал точкой входа для страницы с формой. Он подключает необходимые функции, подготавливает переменные и передаёт их в шаблон.

Файл list.php отвечает за страницу со списком привычек. Он получает данные из data.txt, подготавливает их и подключает шаблон списка.

В результате первый этап был выполнен: HTML-разметка была вынесена в отдельные PHP-шаблоны, а обработка данных осталась в отдельных файлах логики.

Шаг 2. Использование шаблонизатора Twig

На втором этапе в проект был добавлен готовый шаблонизатор Twig.

Для подключения Twig был создан файл composer.json, в котором указана зависимость:

`
{
    "require": {
        "twig/twig": "^3.0"
    }
}
`

После этого была выполнена команда:

`composer install`

Composer установил Twig и создал папку vendor/, в которой находятся файлы библиотеки и автозагрузчик.

Для Twig-версии проекта была создана отдельная папка:

twig_templates/

В неё были помещены шаблоны:

layout.twig
form.twig
list.twig
index.twig

Файл twig_templates/layout.twig содержит общий макет страницы. В нём используется механизм блоков Twig:

`{% block content %}{% endblock %}`

Этот блок является местом, куда дочерние шаблоны вставляют своё содержимое.

Файл twig_templates/index.twig наследует общий макет с помощью конструкции:

`{% extends "layout.twig" %}`

и заполняет блок content:

`{% block content %}
    {% include "form.twig" %}
{% endblock %}`

Для подключения отдельных частей страницы использовалась команда:

`{% include "form.twig" %}`

и аналогично для списка привычек.

Таким образом, в Twig-версии были использованы возможности шаблонизатора:

наследование шаблонов;

блоки;

подключение отдельных шаблонов;

вывод переменных через {{ }};

фильтры.

Для запуска Twig-версии были созданы отдельные файлы:

index_twig.php

list_twig.php

В них подключается автозагрузчик Composer:

`require_once __DIR__ . '/vendor/autoload.php';`

Затем создаётся загрузчик шаблонов:

`$loader = new FilesystemLoader(__DIR__ . '/twig_templates');`

и объект Twig:

`$twig = new Environment($loader);`

После этого данные передаются в Twig-шаблон через метод render():

`echo $twig->render('index.twig', [
    'pageTitle' => 'Трекер привычек',
    'habits' => $habits
]);`

В результате в проекте присутствуют две версии представления:

1. Нативные PHP-шаблоны
   
2. Twig-шаблоны

Это позволяет сравнить два подхода к шаблонизации.

Шаг 3. Дополнительная функция

Для получения максимальной оценки был реализован собственный фильтр Twig.

В проекте был создан фильтр:

`status_icon`

Он используется для отображения статуса привычки в более понятном виде.

Например, в шаблоне Twig можно написать:

`{{ habit.status|status_icon }}`

Фильтр преобразует текстовый статус в значок.
Например:

done → ✅

not_done → ❌

Или похожее отображение, в зависимости от статусов, которые используются в проекте.

Регистрация фильтра выполняется в PHP-файле Twig-версии:

`$twig->addFilter(new TwigFilter('status_icon', function ($status) {
    return match ($status) {
        'done' => '✅',
        'not_done' => '❌',
        default => '➖'
    };
}));`

Этот фильтр решает практическую задачу: делает список привычек более наглядным для пользователя.

Таким образом, дополнительное задание было выполнено: в проекте создан и зарегистрирован собственный фильтр Twig.

Контрольные вопросы:

   1. В чём отличие нативных PHP-шаблонов от шаблонизатора Twig?

Нативные PHP-шаблоны — это обычные .php-файлы, где HTML смешивается с небольшими вставками PHP-кода, а Twig — отдельный шаблонизатор со своим более простым синтаксисом. 
PHP-шаблоны не требуют установки дополнительных библиотек, но в них легче случайно смешать логику и разметку. 
Twig делает шаблоны чище и удобнее, поддерживает наследование, блоки и фильтры, но требует установки через Composer и изучения отдельного синтаксиса.

2. Зачем разделять логику и представление в проекте?

Разделение логики и представления нужно, чтобы код было проще читать, изменять и поддерживать. 
Логика отвечает за обработку данных, проверку формы и сохранение информации, а представление — только за вывод HTML. 
Если смешивать всё в одном файле, проект становится запутанным, сложнее искать ошибки и менять дизайн.

3. Что такое наследование шаблонов в Twig?

Наследование шаблонов в Twig позволяет создать один общий шаблон, например layout.twig, и использовать его на разных страницах. 
Команда {% extends %} указывает, какой шаблон наследуется, а {% block %} определяет часть страницы, которую дочерний шаблон может заменить своим содержимым. 
Это помогает не дублировать одинаковый HTML-код, например шапку, стили и общую структуру страницы.
